# Question Creation Policy

This policy must be used whenever adding, editing or reviewing questions and mark schemes on the site.

The site is for OCR GCSE Computer Science J277.

## General Question Rules

1. All questions must be based on the OCR GCSE Computer Science J277 specification.
2. Do not ask questions that are outside the specification.
3. Questions should be typical of GCSE Computer Science exam style.
4. Questions should be clear, precise and suitable for Year 10 and Year 11 students.
5. Use UK English spelling.
6. Use the word "program" when referring to a computer program.
7. Use OCR-style command words where appropriate, such as:
   - state
   - identify
   - describe
   - explain
   - compare
   - discuss
   - evaluate
   - justify
   - complete
   - write
8. Command words should be bold in the question text using `<strong>`.
9. Do not make the whole question bold.
10. Marks must appear at the end of the question, for example `(3 marks)`.
11. Marks should be allocated based on the number of points or skills expected.
12. One mark should normally equal one valid point, step, reason, comparison or correct part of an answer.
13. Avoid vague questions such as "State one key fact about...".
14. Avoid repeated questions that assess the same idea in almost the same way.
15. Do not create questions simply to reach a fixed number. Quality is more important than quantity.
16. Each topic page should start with a short "What this topic covers" summary of around one or two sentences.

## Subtopic Coverage Rules

1. Each subtopic should have 12 or fewer questions.
2. A subtopic does not have to contain 12 questions.
3. The question set should cover the whole subtopic, not just the easiest parts.
4. Include a sensible range of difficulty:
   - recall questions
   - understanding questions
   - application questions
   - comparison or judgement questions where suitable
5. Use scenarios where possible.
6. Scenarios should be realistic, short and relevant.
7. Avoid long scenarios unless the question genuinely needs one.
8. Do not use the same scenario repeatedly across a topic unless the questions are clearly linked.
9. Do not overload a subtopic with too many low-mark recall questions.
10. Do not include 6 or 8 mark questions unless that style is suitable for the topic.

## Question Type Variety

Use a mixture of question formats where suitable:

1. Text-based written responses.
2. Table completion.
3. Fill in the blanks.
4. Matching terms to descriptions.
5. Multiple-choice style questions where useful.
6. Code completion.
7. Trace table completion.
8. Diagram labelling.
9. Diagram drawing.
10. Scenario-based recommendations.
11. Short comparison questions.
12. Error identification and correction.

When using multiple-choice, matching or option-list questions:

1. Include plausible distractors.
2. Do not make incorrect options obviously wrong.
3. Jumble the order of options.
4. Keep all options within the same topic area.

## Mark Scheme Rules

1. Every question must have a mark scheme.
2. Mark scheme breakdowns should be bulleted.
3. Each bullet should normally represent one creditworthy point.
4. Mark schemes should be clear enough for a student to self-assess.
5. Include "Accept any other valid point" where appropriate.
6. Do not over-credit vague answers.
7. Do not require exact wording unless the exact term is essential.
8. Where examples are acceptable, say so clearly.
9. Where a question asks for a comparison, the mark scheme should credit linked comparative points, not just separate descriptions.
10. Where a question asks students to justify or evaluate, the mark scheme should credit reasoning, not just a stated choice.

## Banded Mark Scheme Rules

1. Any answer worth 6 or more marks must use a banded mark scheme table.
2. The table should describe levels of response.
3. Include a top, middle and lower band where appropriate.
4. The table should include:
   - mark range
   - quality of response
   - expected content
5. The mark scheme may also include indicative content below the banded table.
6. Banded mark schemes should still be student-friendly.
7. Banded mark schemes should not be vague. They must explain what a strong answer includes.

## Programming Question Rules

1. Programming questions must allow either OCR Exam Reference Language or Python.
2. The question text should state that students may use OCR Exam Reference Language or Python.
3. Programming questions should focus on logic, not exact syntax only.
4. Mark schemes should award marks for correct algorithmic steps.
5. Credit should be given for:
   - suitable input
   - suitable output
   - assignment
   - correct variables
   - correct data types where relevant
   - selection
   - iteration
   - conditions
   - list or array use where relevant
   - string handling where relevant
   - correct processing logic
6. Programming mark schemes must include:
   - a bulleted mark breakdown
   - an OCR-style pseudocode example
   - a Python example
7. Python examples should be simple, readable and suitable for GCSE students.
8. Pseudocode should use `=` for assignment.
9. Pseudocode should not use the arrow symbol.
10. Do not force students to use Python only.
11. Do not penalise minor syntax differences if the logic is correct.
12. Do not include advanced programming concepts unless they are intended for the specific topic.

## Writing Programs Mark Rules

1. Questions on 2.6.1 Writing basic programs must be worth between 2 and 4 marks.
2. Questions on 2.6.2 Writing advanced programs must be worth between 5 and 8 marks.
3. Basic program questions should be short and focused.
4. Advanced program questions should require a more complete algorithm or program.
5. Advanced program questions may combine selection, iteration, lists, strings and validation-style logic where suitable.

## Diagram Question Rules

For questions requiring students to draw a diagram, the mark scheme must include the correct diagram or a clear text version of the correct diagram.

This applies to:

1. Network diagrams.
2. Boolean logic diagrams.
3. Flowcharts.
4. Structure diagrams or structure charts.
5. Any other required technical diagram.

Diagram mark schemes should include:

1. The correct drawing or layout.
2. Labels where required.
3. Mark allocation for each correct component.
4. Acceptable alternatives where appropriate.
5. Clear notes about direction of flow or connections where relevant.

## Image Rules

1. Images should be stored in `assets/images/`.
2. Image filenames should be lowercase and descriptive, using hyphens.
3. Do not use vague filenames such as `image1.png` or `diagram.png`.
4. Every image must have suitable alt text.
5. If the image is needed to answer the question, the alt text must describe the important information.
6. Use the reusable `<figure class="question-figure">` pattern.
7. Do not rely on colour alone to communicate meaning.

## SQL Question Rules

1. SQL questions must provide the table or tables needed to answer the question.
2. Tables should include:
   - table name
   - field names
   - sample records where needed
3. If a question involves more than one table, the relationship between the tables should be clear.
4. SQL questions should be appropriate for GCSE level.
5. Mark schemes should credit correct `SELECT`, `FROM`, `WHERE` and any other required clauses.
6. Do not include SQL features beyond the expected GCSE level unless clearly justified by the specification.

## Trace Table Rules

1. Trace table questions must provide the algorithm or code to trace.
2. The number of rows should be manageable for GCSE students.
3. The mark scheme must include the completed trace table.
4. Mark allocation should be clear.
5. If marks are awarded per row or per correct value, state this clearly.

## Fill In The Blank Rules

1. Missing words should test meaningful understanding.
2. Do not remove too many words from one sentence.
3. Provide a word bank if the question would otherwise become too vague.
4. Distractors in a word bank should be plausible and from the same topic area.
5. The mark scheme should show the completed version or list the correct blanks in order.

## Table Completion Rules

1. Tables should be used where they genuinely improve clarity.
2. Table completion questions should have clear column headings.
3. Mark schemes should include the completed table.
4. If multiple answers are acceptable, say so in the mark scheme.

## Scenario Rules

1. Use scenarios where possible.
2. Scenarios should be short, realistic and relevant to the topic.
3. Scenarios should not add unnecessary reading load.
4. Use school, business, home, gaming, media, health, transport or online service contexts where appropriate.
5. The question must require students to apply knowledge to the scenario, not just repeat memorised definitions.

## Difficulty And Mark Balance

1. Most subtopics should include a mixture of 1, 2, 3 and 4 mark questions.
2. Use 5 mark questions only when the answer genuinely needs extended explanation or several linked points.
3. Use 6 or more marks only when a banded response is appropriate.
4. Do not force an 8 mark question into a subtopic unless it is suitable.
5. Avoid making every subtopic follow the same question pattern.
6. Ensure harder questions appear after easier questions where possible.

## OCR Style Rules

1. Use concise exam-style wording.
2. Avoid informal phrasing.
3. Avoid overly leading questions.
4. Avoid asking for content that is not part of the OCR GCSE Computer Science specification.
5. Do not include university-level or industry-level detail unless the GCSE specification requires it.
6. Accept equivalent technical wording where appropriate.

## HTML Formatting Rules

1. Questions and mark schemes must be hard-coded directly inside the topic HTML page.
2. Do not store questions in JavaScript.
3. Do not create topic-specific JavaScript files.
4. Each question must be inside an `<article class="question-card">`.
5. Each question should use `<p class="question-text">`.
6. Each mark scheme must use `<div class="mark-scheme">`.
7. Each reveal button must use `aria-expanded` and `aria-controls`.
8. Each mark scheme must have a unique id.
9. Do not leave empty `<div id="question-list"></div>` containers.
10. Do not use horizontal rules.

## Quality Assurance Checklist

Before committing question changes, check that:

1. Every question is linked to the correct subtopic.
2. Every question is within the OCR GCSE Computer Science specification.
3. The number of marks matches the expected answer.
4. The mark scheme has enough valid points for the marks available.
5. The mark scheme does not award more marks than the question total.
6. Command words are used correctly.
7. Command words are bold and only command words are bold.
8. Mark schemes are hidden by default.
9. Reveal buttons work correctly.
10. There are no duplicate question IDs or mark scheme IDs.
11. There are no duplicate or near-duplicate questions in the same subtopic.
12. Programming questions include both pseudocode and Python examples.
13. Diagram questions include the correct diagram or a clear correct drawing guide.
14. SQL questions include the required table or tables.
15. The page uses UK English spelling.
16. The page contains no placeholder questions.
17. The page contains 12 or fewer questions.
